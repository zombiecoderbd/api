MCP Chat Completion — POST /v1/chat/completions▶

OpenAI-Compatible Chat API

The main chat endpoint. Supports mission mode (multi-agent), single agent mode, and proxy mode (direct provider passthrough).

Request Format

{
  "model": "mission",          // "mission" | agent_id | provider_model_name
  "messages": [
    { "role": "user", "content": "Hello, who are you?" }
  ],
  "stream": false,             // true for SSE streaming
  "temperature": 0.7,
  "session_id": "optional",    // reuse existing session
  "client_id": "my-client",    // client identifier
  "editor": "vscode",          // editor name
  "project_context": "...",    // optional SSOT content
  "tools": [                   // optional MCP tools
    {
      "type": "function",
      "function": {
        "name": "read_file",
        "description": "Read a file",
        "parameters": {
          "type": "object",
          "properties": { "path": { "type": "string" } },
          "required": ["path"]
        }
      }
    }
  ]
}

"code-guru" Single agent mode — runs only that agent with its persona
"deepseek-v4-flash-free" Proxy mode — direct passthrough to provider (OpenCode → Groq → Gemini fallback)

SSE Streaming — Real-time Response▶

How SSE Works

Server-Sent Events (SSE) provide real-time streaming of AI responses. Instead of waiting for the full response, tokens arrive one by one.

Headers Required

Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive

Stream Chunk Format

Each chunk follows OpenAI's format:

{
  "id": "chatcmpl-uuid",
  "object": "chat.completion.chunk",
  "created": 1720000000,
  "model": "mission",
  "choices": [{
    "index": 0,
    "delta": {
      "content": "token text here"    // The actual text chunk
      // OR
      "tool_calls": [...]            // Tool call (if agent uses tools)
    },
    "finish_reason": null            // "stop" on final chunk
  }]
}

JavaScript Client (Fetch API)

async function streamChat(message, model) {
  const response = await fetch('/v1/chat/completions', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      model: model || 'mission',
      messages: [{ role: 'user', content: message }],
      stream: true
    })
  });

  const reader = response.body.getReader();
  const decoder = new TextDecoder();
  let fullText = '';

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    const chunk = decoder.decode(value);
    const lines = chunk.split('\\n');

    for (const line of lines) {
      if (line.startsWith('data: ')) {
        const data = line.slice(6);
        if (data === '[DONE]') break;
        try {
          const parsed = JSON.parse(data);
          const token = parsed.choices?.[0]?.delta?.content || '';
          fullText += token;
          // Update UI with fullText
        } catch (e) {}
      }
    }
  }
  return fullText;
}

MCP Tools — Tool Calls & Usage▶

What are MCP Tools?

Model Context Protocol (MCP) tools allow agents to call external functions during chat. Tools are defined in MCP_TOOLS in hamba.js and passed as function definitions in the chat request.

List Available Tools

# HTTP request
GET /mcp

# Response
{
  "tools": [
    {
      "name": "read_file",
      "description": "Read file content from workspace",
      "params": ["path", "startLine", "endLine"],
      "required": ["path"]
    },
    {
      "name": "write_file",
      "description": "Write content to a file",
      "params": ["path", "content"],
      "required": ["path", "content"]
    }
  ]
}

MCP JSON-RPC 2.0

The MCP endpoint also accepts JSON-RPC 2.0 messages:

# List tools
POST /mcp
{"jsonrpc":"2.0","method":"tools/list","id":1}

# Call a tool
POST /mcp
{"jsonrpc":"2.0","method":"tools/call","params":{"name":"read_file","arguments":{"path":"./hamba.js"}},"id":2}

# Initialize
POST /mcp
{"jsonrpc":"2.0","method":"initialize","params":{"name":"my-client","version":"1.0"},"id":0}

Using Tools in Chat

Pass tool definitions in the chat request. The agent will call tools automatically when needed:

POST /v1/chat/completions
{
  "model": "code-guru",
  "messages": [{"role":"user","content":"Read the hamba.js file and explain the routing"}],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "read_file",
        "description": "Read file content",
        "parameters": {
          "type": "object",
          "properties": {
            "path": {"type": "string", "description": "File path"}
          },
          "required": ["path"]
        }
      }
    }
  ]
}

# Response with tool call
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": null,
      "tool_calls": [{
        "id": "call_abc123",
        "type": "function",
        "function": {
          "name": "read_file",
          "arguments": "{\"path\":\"./hamba.js\"}"
        }
      }]
    },
    "finish_reason": "tool_calls"
  }]
}

WebSocket Tool Calls

Provider System — OpenCode, Groq, Gemini▶

Provider Priority Chain

# Provider Type Models Role
1 OpenCode OpenAI-compatible deepseek-v4-flash-free, mimo-v2.5-free, big-pickle, hy3-free, nemotron-3-ultra-free, north-mini-code-free Primary (free models)
2 Groq OpenAI-compatible llama-3.3-70b, qwen-32b, deepseek-r1, gemma-2, etc. Secondary fallback
3 Gemini Google-native gemini-flash, gemini-lite, gemini-pro Tertiary fallback

Fallback Logic

  1. Try primary provider (OpenCode) with requested model
  2. If rate-limited or errors → try next provider (Groq)
  3. If model not found → try all models from next provider
  4. If all providers fail → return 502 with tried_providers list

Agent Model Masking

Agents like code-guru map to real models (e.g., deepseek-v4-flash-free) but the API hides this. The response shows "model": "code-guru", not the actual model name.

Anti-Dote Type Safety — 6-Step Validation▶

What is Anti-Dote?

A mathematical guarantee system: ∀req ∈ AntidoteRequest: validateSchema(req) ∧ checkProof(req) ⇒ setGoalContract(req) ⇒ execute(req) ⇒ verifyOutput(res)

6 Error Types

Error Code Meaning
INVALID_REQUEST ANTIDOTE_INVALID_REQUEST Malformed input
PROOF_FAILED ANTIDOTE_PROOF_FAILED Logical proof check failed
LIMIT_EXCEEDED ANTIDOTE_LIMIT_EXCEEDED Rate/token/payload limit
CONTRACT_FAILED ANTIDOTE_CONTRACT_FAILED Goal contract not established
EXECUTION_FAILED ANTIDOTE_EXECUTION_FAILED Execution error
VERIFICATION_FAILED ANTIDOTE_VERIFICATION_FAILED Output doesn't satisfy contract

Usage

POST /api/v1/anti-dote
{
  "input": "Analyze this code for security issues",
  "context": "optional system context",
  "tools": [],
  "session_id": "optional",
  "client_id": "optional"
}
